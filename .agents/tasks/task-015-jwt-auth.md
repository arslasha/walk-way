# Task 015: JWT Authentication, Registration & Optional 2FA (Backend)

**Assignee:** Backend Agent
**Status:** To Do

## Context
Для реализации социальных функций, сохранения истории прогулок и персональных папок нам необходимо внедрить полноценную систему пользователей с безопасной авторизацией на бэкенде. Мы будем использовать JWT (JSON Web Tokens) для авторизации клиентов и `django-otp` для опциональной двухфакторной аутентификации (2FA).
- **References:** `PLAN.md` (Phase 3), `docs/prd/phase3-social-accounts.md`.

## Requirements

### 1. Архитектура и База Данных
- Настроить стандартную модель `django.contrib.auth.models.User` (или унаследованную модель, если потребуется).
- Создать модель `UserProfile` (OneToOne с `User`), которая будет содержать:
  - `nickname` (CharField, уникальное, индексируемое, до 50 символов) — публичное имя пользователя.
  - `bio` (TextField, до 500 символов, опционально).
  - `avatar` (ImageField, загрузка в локальную папку `media/avatars/` или облачное хранилище, опционально).
  - `is_2fa_enabled` (BooleanField, по умолчанию `False`).
  - `otp_secret` (CharField, для генерации/хранения секрета TOTP).
  - `backup_codes` (JSONField, для хранения хэшированных кодов восстановления 2FA).

### 2. JWT-эндпоинты (Django REST Framework + SimpleJWT)
Установить и настроить `djangorestframework-simplejwt` со следующими эндпоинтами под префиксом `/api/v1/auth/`:
- `POST /register/` — регистрация нового аккаунта (email, пароль, nickname). Должна создаваться запись `User` и `UserProfile`.
- `POST /login/` — авторизация. Возвращает `access` и `refresh` токены.
  - *Интеграция 2FA:* Если у пользователя включен `is_2fa_enabled: True`, при успешном вводе логина/пароля JWT токены **не выдаются**. Вместо этого возвращается временный сессионный токен 2FA (`pre_auth_token`) и статус `2fa_required: true`.
- `POST /2fa/verify/` — проверка одноразового кода TOTP (или резервного кода). На вход принимает `pre_auth_token` и `code`. При успешной проверке возвращает стандартную пару JWT токенов.
- `POST /refresh/` — стандартное обновление access-токена по refresh-токену.
- `POST /logout/` — инвалидация/добавление refresh-токена в черный список (Blacklist).

### 3. Управление профилем (`/api/v1/profiles/`)
- `GET /profiles/me/` — возвращает полную информацию о текущем пользователе (включая email, nickname, bio, avatar, статус 2FA). Требует авторизации.
- `PATCH /profiles/me/` — обновление профиля (nickname, bio, аватар). При загрузке аватара бэкенд должен:
  - Проверить тип файла (разрешены только JPEG, PNG, WebP) и размер (до 5MB).
  - Автоматически сжать изображение до разрешения $400 \times 400$px с сохранением соотношения сторон перед записью на диск.
- `POST /profiles/me/2fa/enable/` — генерация TOTP-секрета. Возвращает `otp_uri` (для генерации QR-кода на фронтенде) и список из 5 одноразовых кодов восстановления (хэшированных в БД).
- `POST /profiles/me/2fa/confirm/` — подтверждение включения 2FA первым успешным вводом TOTP-кода. Только после этого `is_2fa_enabled` выставляется в `True`.
- `POST /profiles/me/2fa/disable/` — отключение 2FA (требует ввода текущего пароля и TOTP-кода).

## Edge Cases & Error Handling
- **Дублирование nickname:** При попытке зарегистрироваться или обновить никнейм на уже существующий возвращать `400 Bad Request` с сообщением *"Этот никнейм уже занят"*.
- **Обработка невалидных/сломанных изображений:** Если файл аватара поврежден, бэкенд должен вернуть ошибку `400` без падения сервера.
- **Утеря TOTP-устройства:** В случае утери доступа авторизация должна проходить через один из неиспользованных `backup_codes`. После использования код восстановления должен помечаться как использованный (или удаляться).

## Constraints & Rules
- Не хранить пароли или TOTP-секреты в открытом виде (хэшировать с помощью стандартных средств Django/Argon2).
- Время жизни токенов: access-токен — 15 минут, refresh-токен — 7 дней.
- Все изменения в бэкенд-моделях должны сопровождаться генерацией миграций (`python manage.py makemigrations`) внутри Docker-контейнера.
