# PRD: Core Search & Frontend Feed (MVP)

## 1. Контекст и Проблема (Context & Problem)
**Описание фичи:** Базовый поиск и отображение мест (кафе, парки и т.д.) без использования картографического интерфейса и без LLM-генерации вайбов.
**Проблема пользователя:** Пользователям нужно быстро находить интересные места вокруг себя, отфильтрованные по базовым категориям и тегам KudaGo, чтобы решить, куда сходить прямо сейчас.
**Бизнес-ценность:** Это минимально жизнеспособный продукт (MVP). Он докажет, что наша связка Frontend + Backend + PostGIS работает, и позволит начать собирать первый фидбек от пользователей по удобству интерфейса (лента карточек).

## 2. Пользовательские истории (User Stories)
1. **Как пользователь**, я хочу видеть ленту мест (как в Tinder или классической ленте новостей), чтобы быстро оценивать локации визуально.
2. **Как пользователь**, я хочу разрешить доступ к своей геопозиции, чтобы видеть места, которые находятся от меня в пределах определенного радиуса (например, 2 км).
3. **Как пользователь**, я хочу отфильтровать ленту по категориям (например, "Кофейни", "Парки") и базовым тегам KudaGo, чтобы найти то, что мне сейчас нужно.

## 3. Функциональные требования (Functional Requirements)

### Backend (Django REST Framework)
- **Эндпоинт `/api/places/` (GET):**
  - Должен возвращать список мест с пагинацией (LimitOffset или PageNumber).
  - Поддержка параметров фильтрации: `category` (ID или slug), `tags` (список ID), `radius` (в метрах), `lat` и `lon`.
  - Если переданы `lat`, `lon` и `radius`, использовать PostGIS функцию `ST_DWithin` для фильтрации мест в заданном радиусе.
  - Сериализатор должен возвращать координаты места и прикрепленные к нему теги/категории.
- **Эндпоинт `/api/categories/` (GET) и `/api/tags/` (GET):**
  - Возвращают список доступных категорий и тегов для отображения в фильтрах на фронтенде. Возвращать только те теги, у которых `is_vibe=False` (теги KudaGo).

### Frontend (Next.js)
- **UI Компоненты:**
  - **Feed:** Вертикальная лента или сетка карточек мест. Карточка должна показывать: Название, Фото (если есть из KudaGo API), Категорию, Теги, Расстояние (если геопозиция известна).
  - **Filters:** Горизонтальный скролл с "пилюлями" (chips) категорий и кнопка для выбора тегов.
- **Логика:**
  - При первой загрузке запрашивать доступ к `navigator.geolocation`.
  - При изменении фильтров отправлять запрос к `/api/places/` и обновлять стейт ленты.
  - Использовать Server Components где возможно, и Client Components для интерактивных элементов (фильтры, запрос геопозиции).

## 4. Нефункциональные требования (Non-Functional Requirements)
- **Производительность:** Ответ от `/api/places/` при радиусном поиске должен занимать не более 500мс (PostGIS индексы должны быть настроены).
- **Дизайн:** Интерфейс должен соответствовать `DESIGN.md`, создавая "премиальное" ощущение (скругления, тени, отсутствие дефолтных стилей браузера).

## 5. Edge Cases & Error Handling (Крайние случаи)
- **Нет доступа к геопозиции:** Если пользователь запретил доступ, бэкенд должен вернуть список мест без фильтрации по расстоянию (или дефолтный город, например, центр Москвы/Питера). UI должен отобразить уведомление: "Без геопозиции мы показываем популярные места города".
- **Пустой результат:** Если по выбранным фильтрам ничего не найдено, показать Empty State (Скилл `ux-copywriter`: *"Тут пока пустовато. Давай расширим радиус поиска или выберем другие фильтры?"*).
- **Ошибка API:** При падении бэкенда показать Toast/Уведомление (Скилл `ux-copywriter`: *"Кажется, мы потеряли связь. Попробуем еще раз?"*).

## 6. Критерии приемки (Acceptance Criteria)
- [ ] Backend: Созданы API эндпоинты для Places, Categories, Tags.
- [ ] Backend: Гео-фильтрация (ST_DWithin) работает корректно.
- [ ] Frontend: Интерфейс ленты реализован с использованием Tailwind CSS.
- [ ] Frontend: Фильтры категорий и тегов меняют список мест.
- [ ] Frontend: При разрешенной геопозиции места сортируются/фильтруются по расстоянию.
- [ ] Обработаны крайние случаи (нет геопозиции, пустой результат).
