# PRD Generator (Создатель продуктовых требований)

Этот скилл помогает перевести верхнеуровневые идеи из `PLAN.md` в строгие, детальные требования, готовые к передаче разработчику (Архитектору).

## Структура PRD (Product Requirements Document)

При вызове этого скилла, всегда генерируйте документ по следующему шаблону:

### 1. Контекст и Проблема (Context & Problem)
- Краткое описание фичи.
- Какую проблему пользователя она решает? Какова бизнес-ценность?

### 2. Пользовательские истории (User Stories)
Формат: `Как [тип пользователя], я хочу [действие], чтобы [результат/ценность].`
Например: *Как интроверт, я хочу отфильтровать парки с тегом "мало людей", чтобы почитать в тишине.*

### 3. Функциональные требования (Functional Requirements)
Четкий список того, что должна делать система:
- Frontend (Что видит и с чем взаимодействует пользователь).
- Backend (Какие API эндпоинты нужны, какая логика).
- База данных (Нужны ли новые модели или поля).

### 4. Нефункциональные требования (Non-Functional Requirements)
- Производительность (например, "Поиск с учетом геопозиции должен занимать < 500мс").
- Ограничения (например, "Используем только PostGIS для гео-расчетов").

### 5. Edge Cases & Error Handling (Крайние случаи)
- Что делать, если API недоступно?
- Что если по выбранным фильтрам (вайбам) ничего не найдено в радиусе 5 км?
- Что если пользователь запретил доступ к геопозиции?

### 6. Критерии приемки (Acceptance Criteria)
Строгий чеклист (Definition of Done), по которому разработчик поймет, что задача выполнена.

## Процесс работы
1. Спросите пользователя, какую именно фичу из `PLAN.md` (или новую) он хочет детализировать.
2. Сгенерируйте черновик PRD по шаблону выше.
3. Попросите пользователя проверить "Крайние случаи" — не упущено ли что-то важное?
4. После утверждения, сохраните PRD в папку `docs/prd/` (например, `docs/prd/phase2-loop-routing.md`).
