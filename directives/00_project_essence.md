# Directive 00: Project Essence & Domain Knowledge

## 1. The Big Picture (Elevator Pitch)
Мы разрабатываем веб-сервис на стыке гео-справочника и социальной сети. Это не просто "карта с кафешками". Это умный планировщик прогулок и свиданий, который отвечает на вопрос *"Куда пойти и чем заняться, чтобы получить конкретные эмоции?"*. 
Сервис подбирает места и строит маршруты, основываясь не только на физическом расстоянии, но и на "вайбе" (настроении), компании и свободном времени пользователя.

## 2. The Problem We Are Solving
- **Проблема:** Стандартные карты (Google, Yandex) и справочники ищут места по утилитарным признакам (рейтинг, кухня, часы работы). Они не понимают контекста: "я хочу поплакать с подругой", "нужно место для первого свидания без кринжа", "хочу гулять 3 часа, пить кофе и смотреть на архитектуру".
- **Решение:** Глубокая тегизация мест и пространственный (spatial) поиск. Мы даем пользователю поиск по эмоциям и целям, а затем связываем найденные места в удобный маршрут.

## 3. Core Domain Entities (Бизнес-сущности)
При проектировании базы данных (PostgreSQL) и API (Django), всегда держи в уме следующие концепты:

1. **Place (Место):** Базовая единица. Имеет точные гео-координаты. Обрати внимание: в PostGIS необходимо использовать поле типа **`Geography`** (в Django – `models.PointField(geography=True)`), чтобы корректно считать расстояния в метрах. Не содержит фото внутри БД, только URL-ссылки. Имеет внешние идентификаторы (например, `kudago_id`) и флаг `is_active` для механизма «Soft Deletes» (чтобы не ломать исторические маршруты, если место закрылось).
2. **Category (Категория):** Строгая утилитарная классификация из сырых данных источника (например, "Ресторан", "Парк", "Музей").
3. **Vibe / Tag (Вайб/Тег):** Гибкая система эмоциональных атрибутов ("Романтичное", "Дорогое", "Кринж"). Поскольку сырые API (типа KudaGo) не отдают таких вайбов, мы используем механизм **LLM-Enrichment** — автоматическое проставление вайб-тегов на основе анализа описаний заведений через языковые модели. Именно по этим тегам работает основная магия фильтрации.
4. **Route (Маршрут - *Phase 2*):** Ломаная линия (`LineString`), соединяющая выбранные Места. Имеет расчетное время в пути и количество шагов. Тоже хранится в типе `Geography`.
5. **Buffer Zone (Буферная зона - *Phase 2*):** Ключевая фича для поиска Мест вдоль Маршрута с правильным радиусом (например, 200 метров). Для оптимизации используется строго функция `ST_DWithin` с использованием пространственных GiST-индексов (вместо связки `ST_Buffer + ST_Intersects`).
6. **User Context (Пользователь - *Phase 3*):** Социальный слой. Избранные места, списки друзей, возможность поделиться маршрутом в чате.

## 4. The "Killer Features" (Why we use this tech stack)
- **Умная комбинация фильтров:** Пользователь может задать: `[Свидание] + [На природе] + [Не менее 2 часов]`. Наш Django API должен уметь эффективно фильтровать базу, используя `Q` объекты и связи Many-to-Many.
- **Обогащение данных (LLM-Enrichment):** Применение генеративных нейросетей на этапе парсинга для превращения скучных каталожных описаний в живую систему эмоциональных тегов, по которым пользователи хотят искать места.
- **"Заглянуть по пути":** Вместо того чтобы заставлять пользователя отклоняться от маршрута, мы благодаря **PostGIS (`ST_DWithin`)** находим интересные точки ровно вдоль его траектории.

## 6. Architectural Philosophy for the Agent
- **Data First:** В гео-сервисах качество данных важнее красивого UI. Всегда используй метрические координаты (поля `geography=True`) для корректных пространственных расчетов.
- **Idempotency in Execution:** Скрипты парсинга (KudaGo) должны быть готовы к многократному запуску. Используй `update_or_create`, чтобы не плодить дубликаты мест.
- **Soft Deletes:** Воздержись от физического удаления мест (`.delete()`), если они пропали из KudaGo. Используй выключение флага активности `is_active=False`.
- **Fail Gracefully:** Если внешний API (KudaGo) или API LLM-сервиса упали - скрипт парсинга должен логировать ошибку в `stderr` и продолжать работу (или мягко завершаться), а не ронять всю систему.
