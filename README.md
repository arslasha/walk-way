# Геосоциальная информационная система «Walk-Way»

**Walk-Way** — это интеллектуальный планировщик пешеходных прогулок и городских маршрутов, функционирующий как геосоциальный рекомендательный сервис на основе эмоционального контекста («вайбов»), временных ограничений и пространственного анализа.

Проект разработан в рамках курсовой работы по дисциплине «Разработка клиент-серверных приложений».

## 🏛️ Архитектура и ключевые особенности

* **Вайб-ориентированное позиционирование:** Рекомендательная фильтрация мест по эмоционально-семантическим меткам (тегам) с интеллектуальной классификацией на основе описаний.
* **Интерактивная картография:** Высокопроизводительный WebGL-рендеринг векторной карты на базе `MapLibre GL JS` с поддержкой адаптивных стилей и динамического наложения слоев маршрутов.
* **Пешеходная маршрутизация:** Расчет линейных и круговых (кольцевых) маршрутов с использованием секторного алгоритма обхода точек интереса (POI) и интеграцией с OpenRouteService.
* **Анализ геоданных «По пути»:** Пространственные выборки PostGIS (`ST_DWithin`) для поиска и рекомендации мест вдоль построенной геометрии маршрута (в пределах 150-метрового буфера).
* **Ролевая модель доступа (RBAC):** Авторизация на базе JWT-токенов с разграничением прав: обычные пользователи управляют личными коллекциями, модераторы (`is_staff`) могут добавлять, изменять и удалять точки интереса прямо через интерфейс карты и каталога.

## 🛠️ Технологический стек

### Серверная часть (Backend)
* **Ядро:** Python / Django 5.0 (GeoDjango)
* **API:** Django REST Framework 3.15 + `djangorestframework-gis` (GeoJSON сериализация)
* **СУБД:** PostgreSQL 16 с пространственным расширением **PostGIS** 3.4
* **Тестирование:** PyTest (интеграционные тесты API) + Hypothesis (автоматизированное фаззинг-тестирование)
* **Внешние гео-сервисы:** OpenRouteService API (пешеходный профиль)

### Клиентская часть (Frontend)
* **Ядро:** Next.js 14 (App Router, React Server Components)
* **Стилизация:** Tailwind CSS
* **Состояние:** Zustand (с персистентным кэшированием маршрута в localStorage)
* **Картографический движок:** MapLibre GL JS + React Map GL

## 🚀 Локальное развёртывание (Docker)

Для локального запуска системы требуются установленные **Docker** и **Docker Compose**.

### Шаг 1. Подготовка окружения
Клонируйте репозиторий и создайте локальный файл конфигурации `.env` из шаблона:
```bash
git clone https://github.com/arslasha/walk-way.git
cd walk-way
cp .env.example .env
```
*(Параметр `ORS_API_KEY` в `.env` необходим для расчета реальной пешеходной геометрии на карте, при его отсутствии будет активирован прямолинейный fallback-режим).*

### Шаг 2. Оркестрация и запуск контейнеров
Соберите образы и запустите контейнеры базы данных, бэкенда и фронтенда:
```bash
docker compose up -d --build
```
*   **Бэкенд** будет доступен по адресу: `http://localhost:8001/api/v1/`
*   **Фронтенд** будет доступен по адресу: `http://localhost:3001`

### Шаг 3. Инициализация СУБД и импорт геоданных
1. Примените миграции базы данных внутри контейнера бэкенда:
   ```bash
   docker compose exec web python manage.py migrate
   ```
2. Импортируйте исходные точки интереса (POI) Москвы из публичного API KudaGo:
   ```bash
   docker compose exec web python manage.py parse_kudago
   ```
3. Запустите классификатор для генерации вайб-тегов и тем для разговора (icebreakers):
   ```bash
   docker compose exec web python manage.py fill_local_vibes
   ```

### Шаг 4. Запуск автоматических тестов
Запустите интеграционные тесты API и стресс-тесты фаззинг-модуля:
```bash
docker compose exec web pytest places/
```

## ☁️ Облачное развёртывание (Production)

Приложение развёрнуто по методологии 12-factor app в распределенной облачной среде:

1.  **База данных (PostgreSQL + PostGIS):** Развёрнута на PaaS-платформе **Supabase** для обеспечения высокой доступности и эффективного выполнения пространственных запросов.
2.  **Бэкенд (Django REST API):** Контейнеризирован и развёрнут на хостинге **Render** (поддерживает автоматическую сборку из Docker-образа и HTTPS-маршрутизацию).
3.  **Фронтенд (Next.js):** Развёрнут на serverless-платформе **Vercel** с настроенным CI/CD-пайплайном, привязанным к ветке `main` на GitHub.

### Настройка переменных окружения в Production:
Для успешного деплоя бэкенда на Render должны быть заданы следующие переменные:
*   `DATABASE_URL`: Строка подключения к PostgreSQL на Supabase.
*   `DJANGO_SECRET_KEY`: Секретный ключ Django (в режиме `DEBUG=False` проверяется на надежность).
*   `ORS_API_KEY`: Токен доступа к OpenRouteService API.
*   `INTERNAL_API_URL` (для фронтенда): `http://web:8000/api/v1` (адрес внутри сети Docker).
*   `NEXT_PUBLIC_API_URL` (для фронтенда): Внешний адрес бэкенда на Render для браузерных запросов.
