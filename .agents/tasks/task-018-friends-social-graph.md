# Task 018: Social Graph, User Search & Friend Profiles (Backend & Frontend)

**Assignee:** Backend Agent & Frontend Agent (Coordinated)
**Status:** To Do

## Context
Для формирования сообщества пользователям необходимо дать возможность находить друг друга, отправлять запросы в друзья, формировать списки друзей и просматривать их профили с сохраненными публичными папками мест. Это заложит основу для будущих фич совместного real-time планирования маршрутов.
- **References:** `PLAN.md` (Phase 3), `docs/prd/phase3-social-accounts.md`.

## Requirements

### 1. Backend: Модель дружбы и API управления (`/api/v1/friends/`)
- **Модель `Friendship`:**
  - `user_from` (ForeignKey на User, отправитель).
  - `user_to` (ForeignKey на User, получатель).
  - `status` (CharField c выбором: `PENDING` - отправлен, `ACCEPTED` - принят, `BLOCKED` - заблокирован).
  - `created_at` (DateTimeField, авто-создание).
  - *Constraint:* Уникальный индекс на пару (`user_from`, `user_to`), причем пары `(A, B)` и `(B, A)` должны обрабатываться логически как одна связь во избежание дублирования запросов.
- **Эндпоинты API:**
  - `GET /api/v1/friends/` — список принятых друзей текущего авторизованного пользователя.
  - `GET /api/v1/friends/requests/` — список активных входящих и исходящих запросов.
  - `POST /api/v1/friends/request/` — отправить запрос в друзья. Принимает `to_user_id`.
  - `POST /api/v1/friends/respond/` — принять или отклонить запрос. Принимает `friendship_id` и `action` (`accept` / `decline`).
  - `POST /api/v1/friends/block/` — заблокировать пользователя (меняет статус на `BLOCKED`).
- **Публичный профиль другого пользователя (`/api/v1/profiles/{nickname}/`):**
  - Разрешить просмотр профиля другого пользователя.
  - *Правило приватности:* Если пользователи являются **друзьями** (`status: ACCEPTED`), в ответе дополнительно отдавать список их **публичных коллекций** (`is_public: True`) и сохраненных в них мест. Если не друзья — отдавать только базовые данные (аватар, био, ник) без коллекций.

### 2. Frontend: Поиск и Управление друзьями
- **Панель друзей в профиле (`/profile` -> Вкладка "Друзья"):**
  - **Поиск:** Текстовое поле для поиска пользователей по никнейму (`@nickname`). Поиск должен происходить на лету с дебаунсом (300мс).
  - **Список друзей:** Красивый список текущих друзей с аватарами, никнеймами и кнопкой "Удалить из друзей" (с подтверждением).
  - **Счетчик запросов:** Вкладка "Запросы в друзья" с интерактивными кнопками "Принять" (зеленая галочка) и "Отклонить" (серый крестик). При наличии входящих запросов отображать яркий Badge с числом во вкладке.

### 3. Frontend: Публичный профиль друга (`/profile/{nickname}`)
- Создать динамический роут `/profile/[nickname]`.
- При открытии страницы:
  - Запрашивать данные профиля с бэкенда.
  - Если это профиль друга: рендерить аватар, био, никнейм и вкладку "Публичные коллекции", где отображаются его папки мест.
  - Дать пользователю возможность кликнуть на любое место из коллекции друга и добавить его в свой собственный активный маршрут.
  - Если пользователи не друзья — показывать кнопку "Добавить в друзья" (или статус "Запрос отправлен" / "Входящий запрос" с кнопкой "Принять").

## Edge Cases & Error Handling
- **Взаимные запросы:** Если пользователь A отправил запрос B, а затем B пытается отправить запрос A, бэкенд должен автоматически переводить статус существующего запроса в `ACCEPTED` вместо создания новой записи.
- **Дружба с самим собой:** Предотвратить возможность отправить запрос в друзья самому себе как на фронтенде (скрывать кнопку "Добавить"), так и на бэкенде (валидация в сериализаторе с ошибкой `400`).
- **Блокировка (Block):** Если пользователь A заблокировал B, пользователь B не должен иметь возможности отправлять запросы в друзья A или видеть его в результатах поиска.
