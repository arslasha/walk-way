## 1. Сверхважные ограничения и правила (Zero-Agent Rule)

1. **Абсолютная конфиденциальность разработки (Zero-Agent Mention):**
   - **КРИТИЧЕСКИЙ ЗАПРЕТ:** В отчете не должно быть ни малейшего, даже косвенного упоминания использования ИИ-ассистентов, LLM, автоматических агентов, prompt-инжиниринга или сессий генерации. 
   - Все архитектурные решения, написание кода (как Next.js, так и Django), проектирование базы данных PostGIS и проведение автотестов должны быть представлены исключительно как **личная, 100% самостоятельная работа студента-разработчика**.
2. **Научно-технический стиль и глубина:**
   - Никакой «воды», обобщенных фраз или рекламных описаний. Стиль должен быть строго академическим, сухим и технически плотным.
   - Используйте точные термины ГОСТ: «Информационная система», «База данных», «Программный комплекс», «Модуль пространственного анализа», «Клиент-серверное взаимодействие».
3. **Объем и полнота:**
   - Отчет должен детально покрывать каждую строчку архитектуры. Не сокращайте описания таблиц, функций и алгоритмов. Описывайте математику, структуры данных и сетевые взаимодействия во всех подробностях для достижения объема, эквивалентного 50 страницам.
4. **Сквозная нумерация и ГОСТ-нейминг:**
   - Все таблицы и рисунки должны иметь строгую сквозную нумерацию по разделам (например: `Рисунок 1.1 – ...`, `Таблица 2.3 – ...`) в соответствии с примером отчета одногруппника.
   - Перед каждым рисунком или таблицей в тексте **обязательно** должна быть явная ссылка (например: *«...структура базы данных приведена на рисунке 2.1»* или *«...перечень используемых библиотек представлен в таблице 1.2»*).

Перед началом генерации вы обязаны прочитать и проанализировать:

## 3. Требования к контенту: Листинги, Скриншоты и Диаграммы

### А. Листинги кода (Code Listings)
*   Не вставляйте файлы целиком, если они превышают 100 строк. Вместо этого оставляйте наиболее важные фрагменты с подробным математическим или логическим комментарием.
*   Каждый листинг должен оформляться стандартным блоком с указанием языка программирования и сопровождаться подробным разбором алгоритма.
*   Для больших файлов оставляйте блок-инструкцию для студента:
    ```markdown
    > **[ИНСТРУКЦИЯ ДЛЯ СТУДЕНТА]**: Скопируйте в данный листинг полный код файла `backend/places/services.py` с 90 по 186 строки (алгоритм секторной фильтрации).
    ```

### Б. Инструкции по скриншотам (Screenshots Placeholders)
Так как вы генерируете отчет в текстовом формате Markdown, вместо реальных изображений вы должны вставлять структурированный блок-плейсхолдер с **подробнейшей пошаговой инструкцией**, как студент должен сделать этот скриншот.
*   *Пример плейсхолдера:*
    ```markdown
    #### Рисунок 3.2 – Интерфейс карты с построенным круговым маршрутом
    
    > **[ИНСТРУКЦИЯ ПО СОЗДАНИЮ СКРИНШОТА ДЛЯ СТУДЕНТА]**:
    > 1. Запустите проект локально (`npm run dev` во фронтенде, бэкенд в Docker).
    > 2. Перейдите по адресу `http://localhost:3000/map`.
    > 3. Откройте нижнюю панель (Vaul Drawer) или боковую панель.
    > 4. Выберите тип маршрута «Круговой», введите дистанцию «3 км» и нажмите «Рассчитать».
    > 5. Сделайте скриншот экрана браузера так, чтобы была видна оранжевая линия кругового маршрута на темной карте CartoDB и точки интереса (POI) вдоль него.
    > 6. Сохраните скриншот под именем `route_loop_screenshot.png` в папку `docs/screenshots/` и замените этот блок изображением.
    ```

### В. Архитектурные диаграммы (Mermaid)
*   Используйте встроенные диаграммы `mermaid` для визуализации:
    - UML-схемы базы данных (таблицы `Place`, `Category`, `Tag`).
    - Схемы трехслойной архитектуры проекта (Layer 1, Layer 2, Layer 3).

    - Схемы геометрического деления на 3 сектора ($120^\circ$) при генерации круговых маршрутов.

## 4. Структура отчета (ГОСТ-Структурирование)

Отчет должен строго состоять из следующих разделов:

### 1. ТИТУЛЬНЫЙ ЛИСТ И РЕФЕРАТ
*(Плейсхолдеры для заполнения личных данных студента и краткая аннотация проекта на русском языке).*

### 2. СОДЕРЖАНИЕ
*(Автоматически генерируемое оглавление со сквозной нумерацией).*

### 3. ВВЕДЕНИЕ
*   Актуальность разработки геосоциального планировщика пешеходных маршрутов.
*   Объект и предмет исследования.
*   Анализ существующих аналогов (2ГИС, Яндекс.Карты, Komoot) и их недостатков (отсутствие фильтрации по «вайбам»/настроению, слабая интеграция социального контекста для свиданий).
*   Цели и задачи работы.

### 4. ГЛАВА 1. АНАЛИТИЧЕСКИЙ РАЗДЕЛ
*   **1.1. Анализ предметной области:** Особенности пешеходной навигации в мегаполисе, влияние внешних факторов (погода, время суток, бюджет).
*   **1.2. Разработка системных требований:**
    - Функциональные требования (поиск мест, фильтрация, расчет маршрутов, поддержка круговых прогулок).
    - Нефункциональные требования (производительность ГИС-запросов, отзывчивость UI на мобильных устройствах, кросс-браузерность).
*   **1.3. Обоснование выбора стека технологий:**
    - Почему Next.js (App Router, SSR/RSC для SEO и быстрой загрузки).
    - Почему Django + Django REST Framework (быстрая разработка, мощный ORM).
    - Почему PostgreSQL + PostGIS (хранение геопространственных данных в виде `Point` и пространственные индексы `GIST` для `ST_DWithin` запросов).
    - Использование MapLibre GL и свободных векторных тайлов CartoDB.

### 5. ГЛАВА 2. АРХИТЕКТУРА И ПРОЕКТИРОВАНИЕ СИСТЕМЫ
*   **2.1. Концептуальная архитектура ИС:** Описание 3-Layer Architecture (Directives, Orchestration, Deterministic Execution).
*   **2.2. Проектирование схемы базы данных:**
    - Детальное описание сущностей `Place` (с точками PostGIS), `Category`, `Tag` (с полем `is_vibe`).
    - Описание связей, типов данных и наложенных индексов.
    - Вставка UML-диаграммы базы данных (Mermaid).
*   **2.3. Спецификация REST API эндпоинтов:**
    - Детальная документация GET/POST эндпоинтов `/api/v1/places/`, `/api/v1/places/along-route/`, `/api/v1/routes/calculate/` с описанием входных/выходных JSON-структур.
*   **2.4. Алгоритмическое обеспечение:**
    - Математическое описание и геометрия секторного алгоритма круговой маршрутизации (Bearings, разделение на сектора по $120^\circ$, евклидово расстояние на сфере).

### 6. ГЛАВА 3. ТЕХНОЛОГИЧЕСКАЯ РЕАЛИЗАЦИЯ И РАЗРАБОТКА КОДА
*   **3.1. Структура проекта:** Развертывание в Docker-контейнерах, дерево каталогов фронтенда и бэкенда.
*   **3.2. Программная реализация бэкенда:**
    - Разбор работы парсера KudaGo (idempotency, timetable-regex парсер).
    - Код и разбор кастомных фильтров по времени, погоде и цене.
    - Разбор интеграции с внешним API OpenRouteService (ORS) с обработкой таймаутов и лимитов.
*   **3.3. Реализация клиентской части (Фронтенд):**
    - Управление глобальным состоянием маршрутов с помощью Zustand (`routeStore.ts`).
    - Описание логики динамического добавления точек, переопределения порядка (drag-n-drop) и расчета метрик (дистанция, время, шаги).
    - Описание рендеринга слоев карты MapLibre (линия маршрута с эффектом свечения `glow` и точки POI вдоль пути).
    - Описание отзывчивой верстки панели маршрутов и Vaul Bottom Sheet.

### 7. ГЛАВА 4. ТЕСТИРОВАНИЕ И ВЕРИФИКАЦИЯ
*   **4.1. Стратегия тестирования:** Модульное, интеграционное и функциональное тестирование бэкенда и фронтенда.
*   **4.2. Реализация автотестов бэкенда (PyTest):**
    - Описание мокирования внешнего API OpenRouteService для изоляции тестов.
    - Разбор тест-кейсов пространственных запросов (`along-route`) и валидации некорректных входных данных.
    - Результаты прогона тестов (подтверждение успешного прохождения всех 13 тестов).
*   **4.3. Тестирование фронтенда:**
    - Тесты состояния Zustand и синхронизации карты.

### 8. ЗАКЛЮЧЕНИЕ
*   Подведение итогов выполненной работы (сравнение с целями из Введения).
*   Практическая значимость разработанной системы.
*   Перспективы развития (переход к Фазе 3: авторизация, друзья, совместное планирование в реальном времени по WebSockets).
