# Task 012: Backend Advanced Filters (Time, Weather, Budget) & Loop Logic

**Assignee:** Backend Agent
**Status:** Not Started

## Context
В рамках Фазы 2 нам необходимо реализовать продвинутую фильтрацию мест и улучшить логику маршрутизации (включая поддержку круговых маршрутов со стороны бэкенда). Пользователь должен получать рекомендации, актуальные для текущего времени суток (открытые заведения), погодных условий (укрытия в дождь), а также фильтровать места по бюджету (`price_level`).

## Requirements

1. **Обновление модели Place**
   - Добавить поля `opening_hours` (JSONField или строковое поле), `price_level` (целое число или Choices) и `is_indoor` (BooleanField, по умолчанию False).
   - Подготовить и применить миграцию (`makemigrations`, `migrate`).

2. **Обновление KudaGo парсера (Слой 3 - Execution)**
   - Дополнить скрипт парсинга KudaGo для извлечения новых полей (часы работы, цены и, если возможно, определение `is_indoor` на основе тегов/категорий).
   - Убедиться, что парсер остается идемпотентным.

3. **Time-Aware & Weather Filters (API)**
   - В `PlaceViewSet` или соответствующих менеджерах добавить логику фильтрации:
     - Исключение закрытых мест в зависимости от текущего времени пользователя (можно передавать `current_time` в query параметрах).
     - При передаче параметра `weather=rain`, приоритизировать или жестко фильтровать места, где `is_indoor=True`.
     - Добавить фильтр `price_level`.
   
4. **Backend Loop Routing Support**
   - Добавить в `RouteViewSet` или сервисы логику генерации кругового маршрута: при передаче стартовой точки, желаемой дистанции/времени и вайбов, бэкенд должен вернуть набор из 2-3 точек, формирующих круг.

## Constraints & Rules
- Логика пространственных и сложных фильтров должна находиться в моделях/менеджерах/сервисах, а не в самом View.
- Использовать `Q` объекты для сложных логических условий (например, открыто сейчас И (is_indoor ИЛИ погода ясная)).
- Строго соблюдайте 3-Layer Architecture и пишите тесты на новую логику API.
