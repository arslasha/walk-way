1.5 Определение сущностей и ролей предметной области 
На основе проведённого анализа были определены основные сущности 
предметной области и роли пользователей системы. 
Основные сущности системы: 
– Пользователь (User) – зарегистрированный участник системы, 
обладающий определённой ролью и набором прав доступа; 
– Парковка (Parking Lot) – парковочный объект с адресом, 
координатами, описанием и тарифами, принадлежащий определённому 
владельцу; 
– Парковочное место (Parking Spot) – конкретное место на парковке, 
имеющее номер, тип (стандартное, для инвалидов, электро, VIP) и статус 
доступности; 
– Бронирование (Booking) – факт резервирования конкретного 
парковочного места пользователем на определённый период времени; 
– Отзыв (Review) – оценка и текстовый комментарий пользователя о 
парковке. 
Роли пользователей системы: 
– Водитель (user) – основной пользователь системы, осуществляющий 
поиск парковок, бронирование мест, просмотр истории бронирований и 
оставление отзывов; 
– Владелец парковки (owner) – пользователь, управляющий одной или 
несколькими парковками, имеющий возможность добавлять и редактировать 
парковки, просматривать бронирования и статистику; 
– Администратор (admin) – пользователь с полным доступом к системе, 
осуществляющий управление всеми пользователями, модерацию парковок и 
просмотр системной статистики. 
Связи между сущностями: пользователь может владеть несколькими 
парковками (роль owner); парковка содержит множество парковочных мест; 
пользователь может создавать бронирования на парковочные места;

1.6 Функциональные требования

– система должна обеспечивать вход пользователей с использованием

– система должна использовать JWT-токены для аутентификации

Требования к функционалу поиска и бронирования: 
– система должна позволять осуществлять поиск парковок по 
местоположению с отображением результатов на интерактивной карте; 
– система должна предоставлять детальную информацию о парковке:

– система должна проверять отсутствие пересечений бронирований по

– система должна позволять отменять бронирование; 
– система должна автоматически рассчитывать стоимость 
бронирования на основе тарифа и длительности. 
Требования к функционалу управления парковками:

– владелец должен иметь возможность просматривать бронирования

парковок. 
Требования к функционалу администрирования: 
– администратор должен иметь возможность просматривать и

парковки; 
– администратор должен иметь доступ к общей статистике системы. 
Требования к валидации данных: 
– система должна валидировать все входные данные и возвращать 
информативные сообщения об ошибках;

1.7 Нефункциональные требования 
Помимо функциональных требований, к системе предъявляются 
следующие нефункциональные требования: 
– производительность – время отклика API не должно превышать 500 
мс для основных операций; 
– безопасность – пароли пользователей должны храниться в 
хешированном виде (bcrypt), обмен данными должен осуществляться по 
протоколу HTTPS, токены доступа должны иметь ограниченный срок 
действия; 
– масштабируемость – архитектура приложения должна быть stateless, 
что позволяет горизонтальное масштабирование; 
– надёжность – приложение должно корректно обрабатывать ошибки и 
обеспечивать graceful shutdown; 
– удобство использования – интерфейс должен быть адаптивным 
(mobile-friendly) и интуитивно понятным; 
– отказоустойчивость – система должна использовать health check 
эндпоинты для мониторинга состояния; 
– переносимость – приложение должно быть контейнеризировано 
(Docker) для обеспечения единообразной среды развёртывания.

3.4 Аутентификация и авторизация 
Аутентификация реализована на основе JWT-токенов с механизмом 
refresh-токенов. При успешной аутентификации пользователь получает пару 
токенов: access token (срок действия 15 минут) и refresh token (срок действия 7 
дней). Access token используется для аутентификации запросов к защищённым 
эндпоинтам API. При истечении access token клиент автоматически 
запрашивает новую пару токенов, используя refresh token. 
Пароли пользователей хешируются с использованием алгоритма bcrypt 
с параметром cost = 10. В базе данных хранится только хеш пароля, что 
обеспечивает безопасность даже при утечке данных. 
Авторизация реализована на уровне middleware. Middleware requireAuth 
проверяет наличие и валидность JWT-токена. Middleware requireRole 
принимает список разрешённых ролей и проверяет, что роль текущего 
пользователя входит в этот список. Для эндпоинтов, где владелец может 
изменять только свои ресурсы, реализована дополнительная проверка на 
уровне 
handler.

4.2 Fuzz-тестирование 
Fuzz-тестирование (фаззинг) – метод тестирования, при котором на вход 
программе подаются случайные или полуслучайные данные с целью 
обнаружения неожиданного поведения, паник и ошибок обработки. В Go 
фаззинг поддерживается начиная с версии 1.18 через пакет testing. 
В рамках данной работы fuzz-тесты были написаны для следующих 
компонентов: 
– валидация email – проверка, что функция ValidateEmail не вызывает

ЗАКЛЮЧЕНИЕ 
В ходе выполнения курсовой работы был разработан fullstack клиент-
серверный сервис «ParkEase» для поиска и бронирования парковочных мест. 
Все поставленные задачи были выполнены в полном объёме. 
В первой главе проведён анализ предметной области: исследованы 
текущие процессы поиска парковочных мест (AS-IS), выявлены основные 
проблемы, выполнен сравнительный анализ трёх существующих решений-
аналогов (Яндекс.Парковки, Parkopedia, ParkApp), определены сущности и 
роли системы, сформулированы функциональные и нефункциональные 
требования. 
Во второй главе спроектирована клиент-серверная архитектура 
приложения с разделением на фронтенд (Next.js), бэкенд (Go) и базу данных 
(PostgreSQL). Разработана ER-диаграмма из пяти сущностей, спроектирован 
REST API с 15 основными эндпоинтами, применена методология двенадцати 
факторов (12-Factor App). Проектные решения описаны с помощью UML-
диаграмм. 
В третьей главе описана реализация серверной и клиентской частей 
приложения. Реализован полный CRUD для всех сущностей, JWT-
аутентификация с refresh-токенами, ролевая модель с тремя ролями (user, 
owner, admin), валидация входных данных. 
В четвёртой главе представлены результаты тестирования: unit-тесты 
бизнес-логики, fuzz-тесты валидации входных данных (5 модулей), тесты 
ролевой модели на некорректные данные. 
В пятой главе описано развёртывание: контейнеризация через Docker с 
multi-stage сборкой, оркестрация через Docker Compose, автоматический 
деплой на платформу Railway, использование Git/GitHub с стратегией 
ветвления GitHub Flow.
